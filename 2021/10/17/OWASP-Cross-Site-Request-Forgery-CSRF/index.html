概述

Cross-Site Request Forgery (CSRF) is an attack that forces an end user to execute unwanted actions on a web application in which they’re currently authenticated.

一般要结合社会工程学

大多数网站，浏览器在发出请求时都会自动带上认证凭证
网站无法分辨出恶意请求和正常请求

一般指代能够改变服务端状态的CSRF为写类型CSRF
读类型CSRF一般返回不了给攻击者

Login CSRF

让受害者登录攻击者的账号
从而收集受害者在站点的活动记录、敏感信息

stored CSRF flaws

还有一种存储型CSRF
通过img或iframe标签，或者是复杂的XSS攻击把CSRF攻击存储到站点上
一般这种CSRF攻击危害更大
因为受害者更容易浏览攻击点，以及攻击者已经获得了站点的认证

别名

  • XSRF
  • Sea Surf
  • Session Riding（会话劫持）
  • Hostile Linking（恶意链接）
  • (a type of) One-Click attack

无效的防御措施

使用secret cookie

由于发出请求时所有认证凭证都会被提交
所以秘密cookie也会被恶意请求自动带上
包括session标识也是

只接受POST请求

CSRF不一定只能通过恶意链接的方式进行
攻击者也可以通过多种方法自定义恶意的POST请求

Multi-Step Transactions

多步交易、多步操作
攻击者针对每一步编制CSRF攻击即可

URL Rewriting

大概就是把session ID放到URL中，这样攻击者就不能构造恶意链接
然而却引入了另一个漏洞：把session ID在URL中暴露了

使用HTTPS

傻逼吧
HTTPS应当被视为任何有效防御措施的先决条件

下面先简要介绍下防御CSRF攻击的一些原则：

  • 检查并启用框架自带的CSRF防御配置、策略或组件
    • 没有就自己在所有改变状态的请求上加CSRF token，同时做好后端校验
  • 对于有状态（stateful）的软件，使用同步令牌模式（synchronizer token pattern）
  • 对于无状态的软件，使用双重cookie提交（double submit cookies）
  • 在纵深防御策略中至少启用一种CSRF防御措施
    • 针对session cookies的同源cookie特性（SameSite Cookie Attribute），但是注意不要为某个具体的域名设置cookie，否则会引起子域名共享cookie的问题
    • 对某些高敏感操作执行基于用户交互的保护（user interaction based protection）
    • 使用自定义请求头
    • 使用标准headers校验源（verifying the origin with standard headers）
  • XSS攻击可以瓦解任何CSRF防御措施！部署针对XSS的防御
  • 尽量不要使用GET请求去改变服务端状态
