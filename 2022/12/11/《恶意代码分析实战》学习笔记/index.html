第1章 静态分析基础技术

1.1 反病毒引擎扫描

VirusTotal：能够对上传的文件调用多个反病毒引擎来进行扫描

1.2 Hash值

MD5orSHA-1

1.3 查找字符串

查找程序中的所有可打印字符串，能够获得简单的程序功能提示

Windows提供的查找exe程序字符串的命令行工具：String

ASCII编码1个字符占1个字节

Unicode编码1个字符占2个字节

String工具实际上搜索的是连续三个及以上的ASCII/Unicode字符并以NULL为结尾的可打印字符串
意味着找到的不都是真正的字符串
需要留给用户进行过滤甄别

有价值的信息：

  • API函数名
  • 动态链接库名
  • IP地址
  • URL
  • 报错信息（提供大量的复合信息，需要发散侦察）

1.4 加壳与混淆恶意代码

加壳是混淆的一种，加壳后的程序会被压缩且难以分析

使用String搜索出来的字符串很少，说明可能是混淆或加壳的

加壳是混淆程序至少包含LoadLibrary与GetProcAddress函数，用于加载和使用其他函数功能

加壳后的字符串列表、导入表及其他信息都会被压缩

一般使用PEiD工具检测加壳：加壳器类型、编译器类型等

PEiD的一些插件会运行恶意代码

不要轻易在不安全环境下使用PEiD及其插件

PEiD本身也有漏洞（v0.92存在缓冲区溢出漏洞）

需要脱壳才能进一步分析

UPX加壳工具非常流行且容易脱壳，下载UPX工具即可尝试

1.5 PE文件格式

可移植执行（PE）文件格式是Windows可执行文件、对象代码和DLL所使用的标准格式。

PE文件头的信息很有价值

1.6 链接库与函数

导入表、导入函数、代码函数库

通过链接方式被链接到主程序中

PE文件头中的可被找到的信息取决于链接了哪些代码库

1.6.1 静态链接、运行时链接与动态链接

静态链接：整个库的代码都会被复制到可执行程序中 | 很难区分静态链接的代码与程序自身代码

运行时链接：恶意代码常用，当需要使用函数时才链接到库

动态链接：最常见 | 程序启动时链接 | 程序被装载时搜索所需代码库

一些Windows API允许导入没有列出的链接函数
如LoadLibrary、GetProcAddress、LdrGetProcAddress、LdrLoadDll
前两个允许程序访问系统任意库中的任意函数
意味着无法静态分析出程序会使用哪些函数

PE文件头存储了每个将被装载的库文件以及会被使用的函数信息

1.6.2 Dependency Walker，探索动态链接函数

Dependency Walker可以列出可执行文件的动态链接函数

有一个例子，介绍了工具界面及基本使用

应当注意程序是否可能启动其他程序

一些函数命名约定：

  • Ex后缀：经过更新后为了向下兼容
  • A后缀：支持ASCII字符的版本
  • W后缀：支持宽字符的版本

1.6.3 导入函数

微软的MSDN库对Windows API进行了出色的文档化

1.6.4 导入函数

DLL与EXE的导出函数是用来与其他程序或代码进行交互时使用的

PE文件中包含了一个文件中导出了哪些函数的信息，DLL文件中常见

EXE文件中的导出函数会提供有用的信息，例如ServiceMain -> 代码作为服务运行

Dependency Walker工具可查看导出函数

1.7 静态分析技术实践

1.8 PE文件头与分节

PE文件头之后就是分节，文件的一些实际部分，包含了有用的信息

.text：CPU执行指令、其他所有节存储数据和支持性的信息

.rdata：导入导出函数信息、一些只读数据

.data：程序的全局数据

.rsrc：所使用的非可执行资源

1.8.1 使用PEview分析PE文件

展示了PEview的窗口，及一些分析技巧

可以查看编译时间 -> 推测可能的攻击技术新旧程度（但不一定准，且可伪造）

Delphi程序的编译时间都是1992.06.19

IMAGE_OPTIONAL_HEADER的Subsystem指出了程序是CUI还是GUI界面

.text分节中：

Virtual Size：加载后要分配多少内存给这个分节
Size of Raw Data：该分节在磁盘上的原始大小

前者比后者差太远，则意味着可能有加壳

1.8.2 Resource Hacker，查看资源节

可以使用Resource Hacker工具来浏览.rsrc节（这个作者还有一些其他的小工具）

第3章 动态分析基础技术

无论如何，动态分析应该在静态分析之后进行

3.1 沙箱

GFI沙箱能够列出的分析结果：

  • 分析摘要
  • 文件活动
  • 互斥量创建
  • 注册表
  • 网络行为
  • VirusTotal结果

沙箱的缺点：

  • 只能简单运行可执行文件，不能带有命令行选项
  • 不能记录所有事件
  • 有些恶意代码能够检测沙箱环境
  • 有些恶意代码需要系统上有特定的注册表或文件才会执行
  • 恶意DLL的导出函数可能不会被恰当地调用
  • 沙箱的操作系统可能不兼容恶意代码
  • 沙箱不能帮你总结恶意代码的功能/行为

3.2 运行恶意代码

这里主要介绍怎么运行DLL文件

rundll32.exe：所有Windows版本中都有的程序，提供了运行DLL的平台，基础语法如下：
rundll32.exe DLLname, func[ args]
其中参数func的值必须是DLL文件导出函数表中的函数名/序列号（第五个函数为#5）
args接收func需要的参数

一些rundll32.exe的使用例子
rundll32.exe rip.dll, Install：运行Install函数
rundll32.exe rip.dll, #5：运行#5函数
rundll32.exe rip.dll, InstallService Evil：运行InstallService 函数，安装为Evil服务
net start Evil：启动Evil服务
（一些服务函数没有提供安装功能，可能需要手动安装再启动）

也可以通过修改PE头的方法启动DLL
具体为从IMAGE_FILE_HEADER的特征域里擦除IMAGE_FILE_DLL(0x2000)标记
就可以当作是可执行文件执行了

3.3 Process Monitor

进程监视器（Process Monitor）结合并增强了两种工具的功能：
文件监视器（FileMon）和注册表监视器（RegMon）

运行时会监控所有能捕获的系统调用，因此运行时需要限定时间

通过File->Capture Events停止进程监视器捕获事件
通过Edit->Clear Display清除当前捕获事件

以下是运行在Win10上3.91版本的界面
各个条目都可以双击进一步查看详情

和Wireshark类似，会有很多很多信息，需要过滤显示（并不是过滤捕获）
通过Filter->Filter打开过滤器面板，过滤器设置简单易懂
（下面预先定义好的过滤器是默认使用的，一般不显示没什么用的信息）

这里提供了一些按钮，可以很方便地配置一些基本过滤

从左到右依次是：

  • 显示注册表行为
  • 显示文件系统行为
  • 显示网络行为
  • 显示进程行为
  • 显示进程概述（profiling）？

3.4 Process Explorer

进程浏览器（Process Explorer）是动态分析时的任务管理器
可以列出所有活跃进程、载入的DLL、进程属性和整体系统信息
也可以用来kill一个进程、退出用户登录、启动与激活进程

双击能显示详细信息（包括了String，显示一些字符串

但这种方法不一定有效，与攻击者使用的进程替换技术以及内存修改有关（第12章）

能检测本地UDP 53端口，对用户指定的IP地址给出虚假的DNS响应

用法大概就是填入你想重定向DNS查询的IP（默认为网关），选择接口，然后启动服务

监听模式下充当server

4.3 x86体系结构

冯·诺依曼体系结构

x86采用小端字节序（最低位排在第一位/最低的地址上）

寄存器

寄存器可归为以下四类：

  • 通用寄存器：-
  • 段寄存器：定位内存节
  • 状态标志：做出决定
  • 指令指针：定位要执行的下一条指令

EAX通常存储一个函数调用的返回值

x86架构的标志寄存器（EFLAGS）是32位的
一些最重要的标志位介绍：

  • ZF：运算结果为0时置位，否则清除
  • CF：运算结果相对于目标操作数太大/太小时置位，否则清除
  • SF：运算结果最高位为1（复数等情况）时置位，否则清除
  • TF：置位时x86处理器每次只执行一条指令，用于调试

EIP：指令指针，保存了下一条执行指令的内存地址
（重要攻击目标）

简单指令

指令 格式 例子
lea lea destination, source
用于将内存地址（不是值）赋给目的操作数		 lea eax, [ebx+8]
将EBX+8的值赋给EAX
sub - sub eax, 0x10
EAX寄存器值减去0x10
add - add eax, ebx
将EBX值+EAX值的结果保存到EAX
ind - inc edx
EDX值递增1
dec - dec ecx
ECX值递减1
mul mul value
EAX乘上value
结果的高32位放到EDX，低32位放到EAX		 -
imul 有符号版本的mul -
div div value
将EDX与EAX共同存储的一个64位值除以value
结果的商放到EAX，余数放到EDX		 -
idiv 有符号版本的div -
xor - xor eax, eax
将EAX异或EAX的结果放到EAX
（快速将EAX置零）
or - or eax, 0x7575
将EAX与0x7575进行或操作
shr shr destination, count
将目的操作数右移court位
移出边界的位陆续存储到CF标志位中，用0填充新的位		 -
shl shl destination, count
将目的操作数左移court位
移出边界的位陆续存储到CF标志位中，用0填充新的位		 -
ror ror destination, count
将目的操作数循环右移court位
移出边界的位陆续回填到另一端空出来的位		 -
rol rol destination, count
将目的操作数循环左移court位
移出边界的位陆续回填到另一端空出来的位		 -
nop - -

nop指令是无操作，操作码（opcode）是0x90
常用作填充代码（shellcode中的NOP滑板）

栈

函数的内存、局部变量、参数、返回地址、流控制结构等存储在栈中
栈的结构自顶向下

ESP：栈（顶）指针
EBP：栈基址指针（常当作定位器）

函数调用

序言（prologue）：保存函数中要用到的栈和寄存器
结语（epilogue）：将栈和寄存器恢复至函数调用前的状态

leave指令：功能是将ESP等于EBP，以便从栈中弹出EBP
ret指令：从栈中弹出返回地址给EIP

每次函数调用都会生成新的栈帧
函数维护自己的栈帧，直到返回
调用者的栈帧被恢复

mov eax, ss:[esp]：将当前栈顶的值赋值到EAX

指令 说明
pusha 按以下顺序将所有16位寄存器压入栈中
AX、CX、DX、BX、SP、BP、SI、DI
pushad 按以下顺序将所有32位寄存器压入栈中
EAX、ECX、EDX、EBX、ESP、EBP、ESI、EDI
popa 常与pusha结合使用
popad 常与pushad结合使用

上述指令很少被编译器使用
如果看到了，说明可能是手工编写的汇编或者shellcode

条件指令

一般只设置标志位而不改变其使用的操作数

test
功能与and指令一样
test eax, eax常用于检查一个值是否为NULL值

cmp

跳转指令

jmp
jmp location：无条件地跳到目标位置去执行

重复指令

一组操作数据缓冲区的指令

movsx、cmpsx、stosx、scasx
其中x可以是b、w、d，分别代表字节、字、双字

会使用到的寄存器
ESI：源索引
EDI：目的索引
ECX：计数

rep指令会不断增加/减少ESI、EDI并且减少ECX，直到ECX=0
DF标志位指示方向（即是增加还是减少ESI、EDI）：0为加，1为减

指令 说明 例子
movsb 常与rep配套使用
将ESI指向的地址的值赋值到EDI指向的地址		 rep movsb
复制一连串的值
cmpsb 常与repe配套使用
将ESI指向的地址的值减去EDI指向的地址的值（比较）
同时设置相关标志位		 repe cmpsb
找不同的位置
scasb 常与repne配套使用
将ESI指向的地址的值减去AL的值（搜索）
同时设置相关标志位		 repne scasb
搜索首次出现的位置
stosb 常与rep配套使用
将AL的值填入到EDI指向的地址（填充）		 rep stosb
填充一段内存

反汇编窗口的两种模式：窗口模式/文本模式（空格切换）

窗口模式
